# Security Alert List

## 2026-07-22 — 회원가입·프로필 입력 검증 및 XSS

- **어떤 기능에 대한 수정사항인지**: 회원가입, 소개글 수정, 사용자 조회·검색
- **어떤 보안 문제점이었는지**: 사용자명과 소개글에 서버측 길이·문자 검증이 없었고, 중복 폼 필드와 과도하게 긴 검색어를 허용했다. 저장된 소개글이 안전하게 이스케이프되는지 검증하는 회귀 테스트도 없었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.1 서버측 입력 검증** 항목을 기반으로 탐지했다.

- **어떻게 수정했는지**: 사용자명은 공백 제거 후 3~30자의 ASCII 영문·숫자·밑줄·하이픈으로 제한하고 대소문자 비구분 고유 인덱스를 추가했다. 소개글은 500자, 검색어는 허용 문자 30자로 제한하고 모든 검증을 DB 작업 전에 수행하도록 변경했다.

## 2026-07-22 — 비밀번호 저장 및 비밀번호 정책

- **어떤 기능에 대한 수정사항인지**: 회원가입, 로그인, 비밀번호 변경, 기존 계정 마이그레이션
- **어떤 보안 문제점이었는지**: 기존 DB에 평문 비밀번호가 남아 있었고, 해시 비용을 명시적으로 고정하거나 구형 해시를 재해시하지 않았다. 사용자명 포함 비밀번호와 흔한 비밀번호도 허용했다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.3 비밀번호 보안** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 평문 계정 마이그레이션 후 기존 비밀번호 로그인이 유지되는지, 저장값이 `scrypt:32768:8:1` 형식인지, PBKDF2 구형 해시가 로그인 후 재해시되는지, 흔한 비밀번호·사용자명 포함·길이 및 조합 위반 비밀번호가 `400`으로 거부되는지 테스트했다.
- **어떻게 수정했는지**: 앱 초기화 시 비해시 비밀번호를 scrypt로 일괄 변환하고 신규·변경 비밀번호도 동일한 명시적 비용으로 저장한다. 정상 로그인 시 다른 방식·비용의 해시를 재해시하고, 기존 8~12자 조합 정책에 사용자명 포함 및 흔한 비밀번호 금지 규칙을 추가했다.

## 2026-07-22 — 세션·쿠키·CSRF 보호

- **어떤 기능에 대한 수정사항인지**: 로그인 세션, 로그아웃, 비밀번호 변경, HTTP 폼 및 Socket.IO 인증
- **어떤 보안 문제점이었는지**: 로그아웃이 클라이언트 쿠키만 지워 복제된 세션을 서버에서 취소하지 못했고, 비활동·절대 만료가 없었다. 운영 쿠키의 `Secure` 강제와 교차 출처 요청 검증도 없었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.2 CSRF 보호**, **1.4 세션 쿠키 설정**, **1.5 세션 만료 및 재인증** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 세션 토큰 원문이 DB에 저장되지 않는지, 로그인마다 토큰이 회전하는지, 로그아웃이 현재 세션만 제거하는지, 30분 비활동·8시간 절대 만료와 비밀번호 변경 후 모든 HTTP·Socket.IO 세션 무효화를 테스트했다. CSRF 누락·변조·다른 세션 토큰·교차 출처는 거부되고 동일 출처 요청은 성공하는지 검증했으며 운영 쿠키 설정과 `SECRET_KEY` 필수 조건도 별도 프로세스에서 확인했다.
- **어떻게 수정했는지**: 쿠키에는 불투명 세션 토큰만 두고 암호학적으로 생성한 토큰의 SHA-256 해시와 사용자 식별·세션 버전·만료시간은 `user_session`에 저장해 모든 인증 요청에서 서버 상태로 검증한다. 운영 기본값으로 `Secure`, `HttpOnly`, `SameSite=Lax`, host-only 쿠키를 적용하고 운영환경의 `SECRET_KEY`를 필수화했다. 전역 Flask-WTF 검증에 명시적 `Origin` 동일 출처 검증을 추가했다.

## 2026-07-22 — 실패 로그인 제한 및 오류 정보 노출 방지

- **어떤 기능에 대한 수정사항인지**: 로그인, 보안 이벤트 로깅, 공통 오류 응답
- **어떤 보안 문제점이었는지**: 계정·IP별 로그인 시도 제한이 없었고 운영 실행이 `debug=True`로 고정되어 있었다. 상황별 오류 응답과 요청 추적 ID가 없었으며 중복 가입 메시지가 계정 존재 여부를 직접 노출했다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.6 실패 로그인 방어**, **1.7 오류 메시지** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 동일 계정의 분산 IP 5회, 동일 IP의 다수 계정 20회, 병렬 5회 실패가 각각 `429`와 `Retry-After`로 제한되는지 및 15분 후 해제되는지 테스트했다. 존재하지 않는 계정도 더미 해시 검증을 수행하는지, 로그인 로그에 비밀번호·세션 토큰이 없는지, 404·500·503 응답에 예외·경로가 노출되지 않고 요청 ID가 포함되는지, 운영 디버그가 꺼지는지 검증했다.
- **어떻게 수정했는지**: 원문 계정·IP 대신 HMAC 키를 `login_attempt`에 저장해 15분 창의 계정 5회·IP 20회 제한을 원자적으로 적용했다. 존재하지 않는 계정도 더미 scrypt 해시를 검증해 계정 존재 여부에 따른 타이밍 차이를 줄였다. 인증 실패 메시지를 통일하고 중복 가입은 일반 `409`로 처리했다. 운영 디버그를 비활성화하고 `400/403/404/409/429/500/503` 일반 오류 화면, `X-Request-ID`, 민감정보를 포함하지 않는 보안 이벤트 로그를 추가했다.

## 2026-07-22 — 상품 관리 입력·인가·데이터 무결성 강화

- **어떤 기능에 대한 수정사항인지**: 상품 등록, 본인 상품 목록, 상품 수정·삭제, 대표 이미지 업로드·교체·삭제
- **어떤 보안 문제점이었는지**: 상품 이미지의 확장자와 시그니처는 검사했지만 요청 MIME을 검증하지 않았고, 타인 상품과 존재하지 않는 상품의 수정·삭제 응답이 각각 `403`과 `404`로 달라 상품 존재 여부가 노출됐다. 상품 값의 길이·가격·판매자·이미지 파일명 제약이 애플리케이션 검증에만 의존했으며, 동시 수정·삭제 요청이 오래된 데이터로 최신 변경을 덮어쓰거나 삭제할 수 있었다. 이미지 저장 오류를 구체적으로 처리하지 않았고 DB 저장 구간이 광범위한 `Exception`을 처리하고 있었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **2.1 폼 입력 검증**, **2.4 소유자 확인**, **2.5 데이터 무결성**, **5.5 에러 및 예외 처리** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 확장자·MIME·시그니처 불일치와 5MB 초과 이미지가 DB 및 파일을 남기지 않고 `400`으로 거부되는지, 비로그인 생성·수정·삭제가 차단되는지, 서버 관리 필드 조작이 무시되는지, 저장형 XSS가 상품 관련 모든 화면에서 이스케이프되는지 검증했다. 타인 상품과 없는 상품의 수정·삭제가 동일한 `404`인지, 오래된 버전의 수정·삭제가 `409`이며 새 이미지 파일도 정리되는지, DB 직접 입력으로 잘못된 가격·길이·판매자·파일명·버전을 저장할 수 없는지, 유효하지 않은 기존 상품 데이터가 마이그레이션 중 탐지되는지, 파일 저장 및 강제 DB 실패 후 불완전한 상품이나 고아 파일이 남지 않는지 테스트했다. 상품 관리 전용 25개와 전체 48개 테스트가 통과했다.
- **어떻게 수정했는지**: 이미지 확장자별 허용 MIME과 파일 시그니처를 함께 검증하고 파일 저장 실패는 요청 ID를 포함한 보안 로그와 일반화된 `503` 응답으로 처리했다. 소유 상품 조회를 `product_id`와 인증된 `seller_id`가 결합된 쿼리로 제한하고 인가 실패를 `404`로 통일했다. 상품에 `version`을 추가해 수정 시 원자적으로 증가시키고 수정·삭제 쿼리에 예상 버전을 포함하는 낙관적 잠금을 적용했다. 신규 DB에는 `CHECK`·외래키를, 기존 DB에는 동일 검증 트리거와 마이그레이션 전 유효성 검사를 추가해 제목·설명·가격·판매자·이미지 파일명·버전 무결성을 DB에서도 강제했다. DB 예외는 `sqlite3.Error`로 좁히고 실패 시 트랜잭션과 새 이미지 파일을 정리하도록 변경했다.

## 2026-07-22 — 관리자 통신·권한 부여·감사 로그 보안 강화

- **어떤 기능에 대한 수정사항인지**: 관리자 로그인 세션이 사용되는 HTTP 통신, 최초 관리자 계정 승격, 관리자 회원 제재·해제, 상품 숨김·복구, 회원·제재·상품·감사 로그 조회 및 일반 사용자의 관리자 기능 접근 차단
- **어떤 보안 문제점이었는지**: 운영환경에서 `Secure` 쿠키는 사용했지만 애플리케이션이 평문 HTTP 요청을 직접 거부하지 않았고 HSTS가 없어 관리자 세션과 관리 데이터의 전송 보장이 배포 설정에만 의존했다. `ADMIN_USERNAME`이 존재하지 않아도 시작을 계속하여 다른 사용자가 나중에 같은 사용자명을 등록한 뒤 재시작 시 관리자 권한을 받을 가능성이 있었다. 관리자 변경 실패는 감사 로그의 `failure` 결과로 남지 않았고, 관리자 조회 로그는 실제 데이터 조회가 성공하기 전에 `success`로 기록되어 결과가 부정확할 수 있었다. 일반 사용자의 관리자 URL 직접 접근은 차단됐지만 별도의 인가 거부 보안 이벤트가 없었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **5.1 연결 암호화**, **5.2 관리자 활동 로그 관리**, **5.3 관리자 인증 및 인가** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 운영 전송 보호를 활성화했을 때 HTTP 관리자 요청이 `400`으로 거부되고 HTTPS 요청만 성공하며 HSTS와 `Cache-Control: no-store`가 적용되는지 테스트했다. 존재하지 않는 `ADMIN_USERNAME`으로 DB 초기화가 실패하고 권한·감사 데이터가 생성되지 않는지, 정상 승격은 기존 세션을 폐기하고 한 번만 감사되는지 검증했다. 일반 사용자의 관리자 접근이 `403`이며 `admin_authorization_denied` 보안 이벤트를 남기는지, 잘못된 입력·중복 제재·오래된 상품 버전 등 실패한 관리자 변경이 `outcome=failure`로 감사되는지, 감사 저장 실패 시 상태 변경 롤백 또는 민감 조회 `503` 처리가 유지되는지 테스트했다. 관리자 전용 14개와 전체 65개 단위 테스트가 통과했다.
- **어떻게 수정했는지**: 운영환경의 비 HTTPS 요청을 요청 처리 전에 거부하고 모든 운영 응답에 HSTS를 추가했으며 관리자 응답에는 `no-store`와 `no-cache`를 설정했다. TLS 종료 프록시는 정확한 `TRUSTED_PROXY_COUNT`를 설정한 경우에만 전달된 프로토콜과 클라이언트 주소를 신뢰하도록 `ProxyFix`를 제한적으로 적용하고 운영 주의사항을 문서화했다. 설정된 관리자 사용자명이 DB에 없으면 초기화를 실패시켜 사용자명 선점 후 권한 획득 가능성을 차단했다. 관리자 데이터 조회가 성공한 뒤에만 조회 성공 로그를 기록하도록 순서를 변경하고, 검증 실패·보호 대상·중복 처리·없는 대상·동시성 충돌은 구조화된 사유와 HTTP 상태를 포함한 `outcome=failure` 감사 로그로 기록했다. 일반 사용자의 관리자 접근 거부도 요청 ID, 서버가 확인한 사용자 ID와 고정 엔드포인트 이름만 포함하는 보안 이벤트로 기록했다.

- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **3.1 메시지 내용 검증**, **3.2 사용자 인증**, **3.4 Rate Limiting**, **3.5 연결 암호화** 항목을 기반으로 탐지했다. 특히 계정·연결·IP·대화방별 횟수와 바이트 제한, 연결 생성과 동시 연결 상한, 여러 연결·다계정 우회 검증, 운영환경 평문 연결 거부 및 신뢰 프록시 전달 헤더 검증 기준을 적용했다.
- **어떤 테스트를 통해 검증했는지**: 비인증 Socket.IO 연결과 운영환경 평문 연결이 거부되는지, 설정된 프록시 홉의 `X-Forwarded-Proto=https`만 WSS로 인정하고 실제 전달 클라이언트 IP의 HMAC 키를 저장하는지 테스트했다. 사용자당 5개 연결 뒤 추가 연결이 거부되고 종료 후 다시 연결할 수 있는지, 익명 IP 연결 시도가 제한되는지 검증했다. 잘못된 중첩 페이로드도 10회 이후 제한되는지, 여러 계정이 같은 IP에서 보내는 요청이 IP 제한을 공유하는지, 반복된 대용량 페이로드가 바이트 제한에 걸리는지, 제한 로그에 메시지 본문이 포함되지 않는지 테스트했다. 비참여자의 DB 메시지 삽입과 다른 사용자의 대화 기록 조회도 거부되는지 확인했으며 채팅 전용 9개와 전체 79개 단위 테스트가 모두 통과했다.
- **어떻게 수정했는지**: `chat_connection_attempt`, `chat_connection`, `chat_message_attempt` 테이블과 시간순 인덱스를 추가해 제한 상태를 프로세스 전역 객체가 아닌 SQLite에 원자적으로 저장한다. 연결 생성은 1분당 사용자 20회·IP 60회, 활성 연결은 사용자 5개·IP 20개로 제한하고 연결 종료와 만료 시 상태를 정리한다. 유효·무효 메시지 이벤트를 모두 사용자·IP·정규화된 대화쌍별로 계산하여 사용자·대화는 10초 10회 및 5분 100회, IP는 10초 30회 및 5분 300회로 제한하고 각각 별도의 누적 바이트 상한도 적용했다. Socket.IO 입력 버퍼는 16KB, 비동기 이벤트 핸들러는 연결당 직렬 처리로 설정하고 대화 목록 100개·검색 결과 50개·기록 50개 상한을 적용했다. 운영 연결 핸들러에서 보안 전송을 직접 검사하고 `TRUSTED_PROXY_COUNT`가 설정된 경우에만 해당 홉의 `X-Forwarded-Proto`와 `X-Forwarded-For`를 신뢰하도록 수정했다.

- **어떤 보안 문제점이었는지**: 거래 응답의 `no-store` 설정이 주문 조회와 권한 검사를 마친 뒤에 지정되어 비당사자 직접 접근 및 상태 변경 거부 응답에는 일관되게 적용되지 않았고, 기본 `Referrer-Policy`가 같은 출처에는 주문 URL을 전달할 수 있었다. 결제 성공 감사 로그에는 상품 ID와 금액까지 저장되어 감사 목적보다 많은 거래정보가 남았다. 원장 함수는 이미 존재하는 멱등성 키를 찾으면 거래 유형·주문·행위자·차변/대변이 같은 요청인지 확인하지 않고 기존 결과를 반환했으며, 가상 계정의 소유자·종류·생성 시각 변경과 계정 삭제를 막는 DB 불변성 제약이 없었다. 또한 두 구매자의 동시 결제와 구매확정·취소 경합, 같은 멱등성 키 재전송·다른 상품 재사용에 대한 병렬 회귀 테스트가 없어 경쟁 상태 방어의 검증 근거가 부족했다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **8.1 연결 암호화 및 거래정보 비공개**, **8.2 가격·송금액 서버측 입력 검증**, **8.3 경쟁 상태 방지 및 멱등성 보장** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 운영 전송 보호에서 평문 HTTP 거래내역 요청이 `400`으로 거부되고 HTTPS 응답에는 HSTS, `Cache-Control: no-store`, `Referrer-Policy: no-referrer`가 적용되는지 확인했다. 비당사자의 주문 상세·취소 직접 접근이 모두 `404`이며 응답에 주문 ID·상품명·가격·구매자·판매자가 포함되지 않는지, CSRF 누락이 거부되는지 검증했다. 클라이언트가 중복 `price`, 초대형 `amount`, 구매자·판매자·상태를 변조해도 서버의 최신 DB 가격과 인증 세션으로만 주문이 생성되는지 확인했다. 동일 키 재전송은 주문·차감이 한 번만 발생하고 다른 상품이나 다른 원장 의미로 키를 재사용하면 `409` 또는 원장 충돌로 거부되는지 테스트했다. 두 구매자의 병렬 구매와 구매확정·취소 병렬 요청은 각각 하나만 성공하고 주문·결제·상품 상태, 예치 잔액과 원장 거래가 일치하는지 검증했다. 원장 합계가 0인지, 원장 행 수정 및 가상 계정 핵심 필드 변경·삭제가 DB에서 거부되는지도 확인했다. 결제 보안 전용 10개와 전체 회귀 95개 단위 테스트가 모두 통과했다.
- **어떻게 수정했는지**: 모든 거래 라우트 진입 시점에 비공개 응답 표시를 설정해 성공·실패 여부와 무관하게 `no-store`·`no-cache`를 적용하고 거래 응답의 리퍼러 정책을 `no-referrer`로 강화했다. 결제 감사 로그는 결제 대상과 연결하는 최소 주문 ID만 기록하고 상품 ID·금액을 제거했다. 원장 멱등 키가 이미 존재하면 거래 유형, 주문 ID, 생성 행위자와 정렬된 차변·대변 전체가 기존 불변 원장과 정확히 같은 경우에만 재사용하도록 하고 다르면 충돌로 처리했다. `virtual_account`의 식별자·소유자·종류·생성 시각 변경과 삭제를 금지하는 DB 트리거를 추가했으며 기존 명시적 주문 상태 머신, `BEGIN IMMEDIATE`, 버전 조건부 갱신, 상품별 활성 주문 고유 인덱스와 불변 복식부기 원장을 함께 유지했다.

## 2026-07-22 — 상품별 1대1 채팅 컨텍스트 격리 및 브라우저 실행 정책 강화

- **어떤 기능에 대한 수정사항인지**: 상품 상세에서 판매자에게 시작하는 상품별 1대1 채팅, 상품 대화 목록·기록·실시간 수신, 로그인 후 상품 복귀, 채팅 화면의 Socket.IO 클라이언트 로딩
- **어떤 보안 문제점이었는지**: 기존 서버 조회와 브라우저 수신 필터는 사용자 쌍만 대화 경계로 사용하므로 같은 상대와 일반 대화·여러 상품 대화가 공존하면 다른 컨텍스트의 메시지가 현재 화면에 섞일 수 있었다. 멱등 UUID 재사용도 상대만 비교하면 다른 상품 컨텍스트의 저장 결과를 잘못 재사용할 위험이 있었다. 또한 채팅 화면의 외부 Socket.IO 스크립트에 SRI가 없었고 CSP와 클릭재킹 방어 헤더가 없어 외부 리소스 변조 및 실행 가능한 출처를 브라우저가 제한하지 못했다. 로그인 복귀 URL도 새로 도입할 경우 검증 없이 리디렉션하면 오픈 리디렉션이 될 수 있었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **2.2 XSS 방어**, **3.1 메시지 내용 검증**, **3.2 사용자 인증**, **3.3 메시지 검증**, **3.4 Rate Limiting**, **3.5 연결 암호화**, **6.1 SQL 인젝션**, **6.3 보안 헤더 설정**, **7.2 검열된 상품에 대한 인증 및 객체 단위 인가** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 일반 대화와 동일 판매자의 서로 다른 두 상품 대화가 별도 대화 ID와 기록으로 분리되고 현재 메시지 목록에는 선택한 컨텍스트만 표시되는지 확인했다. 조작된 판매자·없는 상품·추가 필드·다른 컨텍스트에서의 동일 멱등 UUID 재사용이 거부되는지, 제3자 소켓에는 이벤트가 전달되지 않는지, 동시 첫 상품 메시지에도 대화가 하나만 생성되는지 검증했다. 숨김·삭제 상품은 신규 사용자가 접근하지 못하고 기존 참여자의 기록은 유지되면서 상품명이 마스킹되는지, DB 트리거가 판매자 없는 상품 대화를 거부하는지 테스트했다. 외부·프로토콜 상대 URL이 로그인 복귀에 사용되지 않는지와 CSP nonce, SRI, `X-Frame-Options`, `no-store` 응답도 검증했다.
- **어떻게 수정했는지**: 대화 식별자를 정렬된 참여자 쌍과 불변 `context_type`·`context_id`의 조합으로 확장하고 상품 대화에 nullable 상품 외래키와 판매자 참여 검증 트리거를 추가했다. 기존 대화는 `direct` 컨텍스트로 메시지와 함께 마이그레이션하며 외래키 무결성을 검사한다. HTTP 기록 조회, Socket.IO 저장·멱등성 확인, 대화별 전송 제한 키와 브라우저 렌더링을 모두 동일한 상품 컨텍스트에 결합하고 메시지는 계속 Jinja 자동 이스케이프와 `textContent`로만 표시한다. 상품 숨김·삭제 후에는 기존 참여자만 대화를 유지하고 상품 메타데이터를 일반 문구로 대체한다. 로그인 `next`는 단일·길이 제한된 동일 출처 상대경로만 허용한다. 요청별 CSP nonce와 제한된 리소스 정책, 클릭재킹·권한 정책 헤더를 추가하고 고정 Socket.IO 파일에 SRI와 `crossorigin`을 적용했다.
