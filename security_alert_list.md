# Security Alert List

## 2026-07-22 — 회원가입·프로필 입력 검증 및 XSS

- **어떤 기능에 대한 수정사항인지**: 회원가입, 소개글 수정, 사용자 조회·검색
- **어떤 보안 문제점이었는지**: 사용자명과 소개글에 서버측 길이·문자 검증이 없었고, 중복 폼 필드와 과도하게 긴 검색어를 허용했다. 저장된 소개글이 안전하게 이스케이프되는지 검증하는 회귀 테스트도 없었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.1 서버측 입력 검증** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 빈 값·길이 초과·허용되지 않은 문자·중복 필드가 `400`이며 DB를 변경하지 않는지, 추가 권한 필드가 무시되는지, 501자 소개글이 거부되는지, 저장된 `<script>`가 사용자 목록에서 HTML 이스케이프되는지 테스트했다.
- **어떻게 수정했는지**: 사용자명은 공백 제거 후 3~30자의 ASCII 영문·숫자·밑줄·하이픈으로 제한하고 대소문자 비구분 고유 인덱스를 추가했다. 소개글은 500자, 검색어는 허용 문자 30자로 제한하고 모든 검증을 DB 작업 전에 수행하도록 변경했다.

## 2026-07-22 — 비밀번호 저장 및 비밀번호 정책

- **어떤 기능에 대한 수정사항인지**: 회원가입, 로그인, 비밀번호 변경, 기존 계정 마이그레이션
- **어떤 보안 문제점이었는지**: 기존 DB에 평문 비밀번호가 남아 있었고, 해시 비용을 명시적으로 고정하거나 구형 해시를 재해시하지 않았다. 사용자명 포함 비밀번호와 흔한 비밀번호도 허용했다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.3 비밀번호 보안** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 평문 계정 마이그레이션 후 기존 비밀번호 로그인이 유지되는지, 저장값이 `scrypt:32768:8:1` 형식인지, PBKDF2 구형 해시가 로그인 후 재해시되는지, 흔한 비밀번호·사용자명 포함·길이 및 조합 위반 비밀번호가 `400`으로 거부되는지 테스트했다.
- **어떻게 수정했는지**: 앱 초기화 시 비해시 비밀번호를 scrypt로 일괄 변환하고 신규·변경 비밀번호도 동일한 명시적 비용으로 저장한다. 정상 로그인 시 다른 방식·비용의 해시를 재해시하고, 기존 8~12자 조합 정책에 사용자명 포함 및 흔한 비밀번호 금지 규칙을 추가했다.

## 2026-07-22 — 세션·쿠키·CSRF 보호

- **어떤 기능에 대한 수정사항인지**: 로그인 세션, 로그아웃, 비밀번호 변경, HTTP 폼 및 Socket.IO 인증
- **어떤 보안 문제점이었는지**: 로그아웃이 클라이언트 쿠키만 지워 복제된 세션을 서버에서 취소하지 못했고, 비활동·절대 만료가 없었다. 운영 쿠키의 `Secure` 강제와 교차 출처 요청 검증도 없었다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.2 CSRF 보호**, **1.4 세션 쿠키 설정**, **1.5 세션 만료 및 재인증** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 세션 토큰 원문이 DB에 저장되지 않는지, 로그인마다 토큰이 회전하는지, 로그아웃이 현재 세션만 제거하는지, 30분 비활동·8시간 절대 만료와 비밀번호 변경 후 모든 HTTP·Socket.IO 세션 무효화를 테스트했다. CSRF 누락·변조·다른 세션 토큰·교차 출처는 거부되고 동일 출처 요청은 성공하는지 검증했으며 운영 쿠키 설정과 `SECRET_KEY` 필수 조건도 별도 프로세스에서 확인했다.
- **어떻게 수정했는지**: 쿠키에는 불투명 세션 토큰만 두고 암호학적으로 생성한 토큰의 SHA-256 해시와 사용자 식별·세션 버전·만료시간은 `user_session`에 저장해 모든 인증 요청에서 서버 상태로 검증한다. 운영 기본값으로 `Secure`, `HttpOnly`, `SameSite=Lax`, host-only 쿠키를 적용하고 운영환경의 `SECRET_KEY`를 필수화했다. 전역 Flask-WTF 검증에 명시적 `Origin` 동일 출처 검증을 추가했다.

## 2026-07-22 — 실패 로그인 제한 및 오류 정보 노출 방지

- **어떤 기능에 대한 수정사항인지**: 로그인, 보안 이벤트 로깅, 공통 오류 응답
- **어떤 보안 문제점이었는지**: 계정·IP별 로그인 시도 제한이 없었고 운영 실행이 `debug=True`로 고정되어 있었다. 상황별 오류 응답과 요청 추적 ID가 없었으며 중복 가입 메시지가 계정 존재 여부를 직접 노출했다.
- **어떤 체크리스트를 기반으로 탐지했는지**: `feature_security_checklist.md`의 **1.6 실패 로그인 방어**, **1.7 오류 메시지** 항목을 기반으로 탐지했다.
- **어떤 테스트를 통해 검증했는지**: 동일 계정의 분산 IP 5회, 동일 IP의 다수 계정 20회, 병렬 5회 실패가 각각 `429`와 `Retry-After`로 제한되는지 및 15분 후 해제되는지 테스트했다. 존재하지 않는 계정도 더미 해시 검증을 수행하는지, 로그인 로그에 비밀번호·세션 토큰이 없는지, 404·500·503 응답에 예외·경로가 노출되지 않고 요청 ID가 포함되는지, 운영 디버그가 꺼지는지 검증했다.
- **어떻게 수정했는지**: 원문 계정·IP 대신 HMAC 키를 `login_attempt`에 저장해 15분 창의 계정 5회·IP 20회 제한을 원자적으로 적용했다. 존재하지 않는 계정도 더미 scrypt 해시를 검증해 계정 존재 여부에 따른 타이밍 차이를 줄였다. 인증 실패 메시지를 통일하고 중복 가입은 일반 `409`로 처리했다. 운영 디버그를 비활성화하고 `400/403/404/409/429/500/503` 일반 오류 화면, `X-Request-ID`, 민감정보를 포함하지 않는 보안 이벤트 로그를 추가했다.
