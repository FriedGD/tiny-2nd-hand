# Python 시큐어코딩 보안 체크리스트

이 체크리스트는 `Python_시큐어코딩_가이드(2023년_개정본).pdf`의 본문과 부록을 바탕으로 Python 코드의 보안약점을 점검할 수 있도록 정리한 것이다.

가이드는 구현단계 보안약점 49개 중 Python 언어 특성상 다음 3개 항목을 제외한 46개 항목을 다룬다.

- 메모리 버퍼 오버플로우
- 해제된 자원 사용
- 초기화되지 않은 변수 사용

## 1. 입력데이터 검증 및 표현 - 16개

- [ ] **SQL 삽입**: 외부 입력값을 SQL 문자열 결합, f-string, `%`, `format()`으로 조립하지 않고 파라미터 바인딩이나 ORM을 사용하는가?
- [ ] **코드 삽입**: 신뢰할 수 없는 값이 `eval()`, `exec()`, `compile()` 또는 동적 import에 전달되지 않는가?
- [ ] **경로 조작 및 자원 삽입**: 파일명, 경로, 포트 등은 허용목록으로 검증하고 정규화된 최종 경로가 허용 디렉터리 내부인지 확인하는가?
- [ ] **크로스사이트 스크립트(XSS)**: 출력 컨텍스트에 맞는 이스케이프를 적용하고 Django/Jinja 자동 이스케이프를 비활성화하지 않았는가? `mark_safe`, `safe`, 직접 HTML 조립을 검토했는가?
- [ ] **운영체제 명령어 삽입**: `os.system()`과 `shell=True` 사용을 피하고, 고정 명령과 분리된 인자 배열을 사용하는가?
- [ ] **위험한 형식 파일 업로드**: 확장자, MIME, 실제 파일 형식, 크기, 개수를 검증하고 무작위 파일명과 최소 권한으로 웹 루트 밖에 저장하는가?
- [ ] **신뢰되지 않은 URL 주소로 자동접속 연결**: 리다이렉트 대상은 서버 측 허용목록 또는 검증된 상대 URL로 제한하는가?
- [ ] **부적절한 XML 외부 개체 참조(XXE)**: XML의 DTD와 외부 엔티티 처리를 비활성화했는가? `lxml`이라면 `resolve_entities=False`, `no_network=True` 등을 확인했는가?
- [ ] **XML 삽입**: XPath/XQuery를 문자열 결합으로 만들지 않고 변수 바인딩 또는 특수문자 검증을 사용하는가?
- [ ] **LDAP 삽입**: LDAP 필터 값을 전용 이스케이프 함수로 처리하고 허용목록 검증과 최소 권한 바인딩 계정을 사용하는가?
- [ ] **크로스사이트 요청 위조(CSRF)**: 상태 변경 요청에 프레임워크의 CSRF 토큰 검증이 적용되고 `csrf_exempt` 같은 우회 설정이 없는가?
- [ ] **서버사이드 요청 위조(SSRF)**: 서버가 요청할 URL, 스킴, 호스트, 포트를 허용목록으로 제한하고 내부망, 루프백, 메타데이터 주소 및 리다이렉트 우회를 차단하는가?
- [ ] **HTTP 응답분할**: 사용자 입력이 헤더, 쿠키, 로그에 들어갈 때 CR/LF를 거부하고 헤더 값도 허용목록으로 제한하는가?
- [ ] **정수형 오버플로우**: NumPy나 C 확장 등의 고정 길이 정수 사용 전후에 최소값과 최대값 범위를 검사하는가?
- [ ] **보안기능 결정에 사용되는 부적절한 입력값**: 권한, 역할, 가격 등 보안 결정이 쿠키, hidden field, 요청 파라미터 같은 클라이언트 값에 의존하지 않고 서버 상태를 기준으로 수행되는가?
- [ ] **포맷 스트링 삽입**: 사용자 입력 자체가 `.format()` 등의 포맷 문자열이 되지 않고 고정 포맷의 데이터 인자로만 들어가는가?

## 2. 보안기능 - 16개

- [ ] **적절한 인증 없는 중요 기능 허용**: 패스워드나 개인정보 변경 등 중요 작업에 서버 측 인증과 필요시 재인증을 요구하는가?
- [ ] **부적절한 인가**: 모든 실행 경로에서 사용자 역할뿐 아니라 대상 자원의 소유권까지 확인하는가?
- [ ] **중요한 자원에 대한 잘못된 권한 설정**: 설정, 키, 업로드, 로그 파일에 최소 권한을 부여하고 `0o777` 같은 과도한 권한을 사용하지 않는가?
- [ ] **취약한 암호화 알고리즘 사용**: DES, RC4, MD5, SHA-1 등 취약한 알고리즘이나 자체 제작 암호를 사용하지 않는가?
- [ ] **암호화되지 않은 중요정보**: 개인정보, 금융정보, 인증정보를 평문 저장 또는 전송하지 않고 TLS와 안전한 저장 암호화를 사용하는가?
- [ ] **하드코드된 중요정보**: 패스워드, API 키, 토큰, 개인키가 코드, 설정 기본값, 테스트 파일에 포함되지 않는가?
- [ ] **충분하지 않은 키 길이 사용**: 가이드 기준으로 RSA/DSA 2,048비트 이상, ECC 224비트 이상, 대칭키 128비트 이상인가?
- [ ] **적절하지 않은 난수 값 사용**: 토큰, OTP, 키, 세션 ID에 `random` 대신 `secrets`, `os.urandom()` 등 암호학적 난수를 사용하는가?
- [ ] **취약한 패스워드 허용**: 충분한 길이와 금지 패스워드 정책을 적용하고 사용자 ID 등 쉽게 추측할 수 있는 값을 허용하지 않는가?
- [ ] **부적절한 전자서명 확인**: 중요한 데이터, 코드, 다운로드 파일은 신뢰된 공개키로 서명을 검증한 후 사용하는가?
- [ ] **부적절한 인증서 유효성 검증**: `verify=False`, `CERT_NONE`, 호스트명 검사 비활성화가 없고 신뢰 체인과 호스트명을 검증하는가?
- [ ] **사용자 하드디스크에 저장되는 쿠키를 통한 정보 노출**: 중요정보를 영속 쿠키에 저장하지 않고 만료시간을 최소화하며 `Secure`, `HttpOnly`, 적절한 `SameSite`를 설정하는가?
- [ ] **주석문 안에 포함된 시스템 주요정보**: 주석, 예제, TODO에 계정, 패스워드, 내부 주소, 키가 남아 있지 않은가?
- [ ] **솔트 없이 일방향 해시 함수 사용**: 사용자별 고유 솔트와 반복 비용을 사용하는가? 단순 SHA-256보다 Argon2, bcrypt, scrypt, PBKDF2 같은 패스워드 전용 함수를 우선하는가?
- [ ] **무결성 검사 없는 코드 다운로드**: 신뢰된 출처와 HTTPS를 사용하고 서명 또는 신뢰된 해시를 검증한 뒤 최소 권한으로 실행하는가?
- [ ] **반복된 인증시도 제한 기능 부재**: 계정 및 IP별 속도 제한, 지연, 잠금, CAPTCHA 또는 추가 인증을 적용하고 모니터링하는가?

## 4. 에러처리 - 3개

- [ ] **오류 메시지 정보노출**: 사용자 응답에 스택 트레이스, 경로, SQL, 설정, 키가 노출되지 않고 운영환경의 `DEBUG`가 꺼져 있는가?
- [ ] **오류상황 대응 부재**: 예상 가능한 실패를 처리하며 예외를 잡은 뒤 `pass`로 무시하거나 취약한 기본값으로 계속 실행하지 않는가?
- [ ] **부적절한 예외 처리**: 반환값을 검사하고 bare `except:` 또는 지나치게 넓은 `except Exception` 대신 구체적인 예외를 처리하는가?

## 5. 코드오류 - 3개

- [ ] **Null Pointer 역참조에 해당하는 `None` 참조**: `None`이 가능한 반환값을 사용하기 전에 명시적으로 확인하고 `0`, 빈 문자열과 혼동하지 않는가?
- [ ] **부적절한 자원 해제**: 파일, 소켓, DB 연결, 락을 `with` 컨텍스트 관리자나 `finally`로 항상 해제하는가?
- [ ] **신뢰할 수 없는 데이터의 역직렬화**: 외부 데이터에 `pickle.loads()`, `yaml.load()` 등 코드 실행 가능한 역직렬화를 사용하지 않는가? 필요하면 JSON과 스키마 검증을 사용하는가?

## 6. 캡슐화 - 4개

- [ ] **잘못된 세션에 의한 데이터 정보 노출**: 클래스 변수, 전역 변수, 싱글톤에 요청별 사용자 데이터를 저장하지 않는가?
- [ ] **제거되지 않고 남은 디버그 코드**: 운영 배포에서 Django/Flask 디버그 모드, 테스트 엔드포인트, 우회 계정, 진단 출력이 제거됐는가?
- [ ] **Public 메서드로부터 반환된 Private 배열**: 내부 리스트나 딕셔너리를 그대로 반환하지 않고 필요한 경우 복사본을 반환하는가?
- [ ] **Private 배열에 Public 데이터 할당**: 호출자가 전달한 리스트나 딕셔너리의 참조를 내부 상태로 그대로 저장하지 않고 복사 및 검증하는가?

## 7. API 오용 - 2개

- [ ] **취약한 API 사용**: 패키지의 출처, 유지보수 상태, 알려진 CVE, 버전, 의존성을 검토하고 버전을 고정하며 SBOM과 업데이트 절차를 운영하는가?

## 코드 리뷰 적용 방법

1. HTTP 요청, CLI 인자, 환경변수, 파일, 메시지 큐, DB 조회 결과 등 외부 데이터의 유입 지점(source)을 찾는다.
2. 유입된 데이터가 SQL, 운영체제 명령, 파일 경로, HTML, URL, XML, LDAP, 역직렬화, 인증 및 인가 결정 같은 위험한 사용 지점(sink)에 도달하는지 추적한다.
3. source와 sink 사이에 허용목록 검증, 타입 및 범위 검증, 안전한 API, 출력 인코딩, 권한 확인이 존재하는지 확인한다.
4. 검증 로직이 클라이언트가 아닌 서버에서 실행되며 모든 실행 경로에 일관되게 적용되는지 확인한다.
5. 보안 통제를 우회하는 예외처리, 디버그 설정, 테스트 코드, 기능 플래그가 운영환경에 남아 있지 않은지 확인한다.
