# 기능별 Python 보안 점검 체크리스트

이 문서는 `secure_coding_checklist.csv`의 27개 기능별 점검 항목을 `security_checklist.md`의 Python 시큐어코딩 기준으로 구체화한 것이다.

## 사용 방법

- 각 항목의 모든 세부 체크박스를 코드, 설정, 테스트 결과로 확인한다.
- 일부 항목이 해당 기능에 적용되지 않으면 체크 대신 `N/A`와 사유를 기록한다.
- 클라이언트 검증은 사용성을 위한 보조 수단으로만 인정하며, 보안 통과 여부는 서버 측 검증을 기준으로 판단한다.
- 인증과 인가는 별개로 확인한다. 로그인 여부뿐 아니라 요청한 자원에 대한 소유권과 역할도 검사해야 한다.
- 검증 실패 시 서버는 요청을 거부하고 데이터 변경이나 후속 처리를 수행하지 않아야 한다.

---

## 1. 회원가입 및 프로필 관리

### 1.1 서버측 입력 검증

연계 기준: 코드 삽입, XSS, 보안기능 결정에 사용되는 부적절한 입력값, 포맷 스트링 삽입

- [ ] 회원가입 및 프로필 API가 `request.form`, `request.json`, 경로·쿼리 파라미터를 서버에서 다시 검증한다.
- [ ] `username`은 문자열 타입, 필수 여부, 최소·최대 길이, 허용 문자와 정규화 정책을 검사한다. 예를 들어 3~30자와 프로젝트가 허용한 문자만 받는다.
- [ ] 이메일, 전화번호, URL 등 형식이 있는 값은 전용 파서 또는 검증기를 사용하며 정규식 하나만으로 신뢰하지 않는다.
- [ ] 역할, 활성 상태, 관리자 여부, 가입 등급 같은 보안 속성은 요청 본문의 값을 그대로 모델에 대입하지 않고 서버가 결정한다.
- [ ] 예상하지 않은 필드는 거부하거나 명시적으로 무시하며 `Model(**request.json)` 같은 대량 할당으로 권한 필드가 변경되지 않는다.
- [ ] 사용자 입력을 `eval()`, `exec()`, `compile()` 또는 사용자 제어 포맷 문자열에 전달하지 않는다.
- [ ] 저장된 사용자 입력을 HTML에 표시할 때 템플릿 자동 이스케이프를 유지하고 `|safe`, `mark_safe`, 직접 HTML 문자열 조립을 사용하지 않는다.
- [ ] 잘못된 타입, 빈 값, 최대 길이 초과, 허용되지 않은 문자 및 추가 필드에 대한 실패 테스트가 있으며 서버가 4xx로 거부한다.

통과 기준: 유효하지 않은 입력은 DB 작업 전에 거부되고, 클라이언트가 역할 등 서버 관리 필드를 조작할 수 없으며, 출력 시 스크립트가 실행되지 않는다.

### 1.2 CSRF 보호

연계 기준: 크로스사이트 요청 위조(CSRF)

- [ ] 회원가입, 로그인, 로그아웃, 프로필 수정, 이메일·비밀번호 변경 등 상태 변경 요청에 프레임워크의 CSRF 보호가 적용된다.
- [ ] 쿠키 기반 인증을 사용하는 모든 `POST`, `PUT`, `PATCH`, `DELETE` 요청에서 세션과 연결된 CSRF 토큰을 검증한다.
- [ ] Django의 `CsrfViewMiddleware`, Flask-WTF 또는 동등한 보호 기능이 전역적으로 활성화돼 있다.
- [ ] `csrf_exempt`와 같은 예외는 제거돼 있거나 불가피한 경우 대상, 이유, 대체 보호 수단이 문서화돼 있다.
- [ ] 상태 변경을 `GET` 요청으로 수행하지 않는다.
- [ ] 토큰 누락, 변조, 다른 세션의 토큰 및 허용되지 않은 `Origin` 요청이 403 등으로 거부되는 테스트가 있다.

통과 기준: 공격자가 사용자의 인증 쿠키만 이용해 계정이나 프로필 상태를 변경할 수 없다.

### 1.3 비밀번호 보안

연계 기준: 암호화되지 않은 중요정보, 취약한 암호화 알고리즘, 솔트 없이 일방향 해시 함수 사용, 취약한 패스워드 허용

- [ ] 비밀번호는 평문, 복호화 가능한 암호문, Base64 또는 단순 SHA-256/MD5 해시로 저장하지 않는다.
- [ ] Argon2, bcrypt, scrypt 또는 프레임워크가 제공하는 PBKDF2 기반 비밀번호 해시 함수를 사용한다.
- [ ] 라이브러리가 사용자별 고유 솔트를 안전하게 생성하며, 직접 고정 솔트를 구현하지 않는다.
- [ ] 해시 비용 파라미터는 운영환경에서 허용 가능한 범위 내의 충분한 비용으로 설정하고 구형 해시는 로그인 성공 시 재해시한다.
- [ ] 비밀번호에는 프로젝트 정책에 따른 충분한 최소 길이를 적용하고 사용자명, 흔한 비밀번호, 유출된 비밀번호 등 쉽게 추측 가능한 값을 제한한다.
- [ ] 비밀번호 확인은 프레임워크의 검증 함수 또는 상수 시간 비교 함수를 사용한다.
- [ ] 비밀번호와 재설정 토큰이 로그, 예외 메시지, 분석 이벤트 또는 응답에 기록되지 않는다.

통과 기준: DB가 유출돼도 비밀번호 원문을 즉시 복원하기 어렵고, 애플리케이션 어디에도 평문 비밀번호가 지속적으로 남지 않는다.

### 1.4 세션 쿠키 설정

연계 기준: 사용자 하드디스크에 저장되는 쿠키를 통한 정보 노출, 적절하지 않은 난수 값 사용

- [ ] 운영환경 세션 쿠키에 `Secure=True`, `HttpOnly=True`, 적절한 `SameSite` 값이 설정돼 있다.
- [ ] 쿠키의 `Domain`과 `Path` 범위가 기능에 필요한 최소 범위로 제한돼 있다.
- [ ] 세션 쿠키에 비밀번호, 개인정보, 권한 등급 또는 신뢰할 수 있는 보안결정 값을 평문으로 저장하지 않는다.
- [ ] 세션 ID는 `random`이 아닌 프레임워크 또는 `secrets` 기반의 예측 불가능한 값이다.
- [ ] 세션 쿠키에 불필요하게 긴 `Max-Age` 또는 `Expires`를 설정하지 않는다.
- [ ] 개발용 HTTP 설정이 운영환경에서 `Secure=False`로 배포되지 않도록 환경별 설정과 테스트가 있다.

통과 기준: JavaScript와 평문 HTTP를 통한 세션 쿠키 탈취 위험이 제한되고, 쿠키 변조만으로 권한이 변경되지 않는다.

### 1.5 세션 만료 및 재인증

연계 기준: 적절한 인증 없는 중요 기능 허용, 보안기능 결정에 사용되는 부적절한 입력값

- [ ] 서버가 비활동 만료시간과 절대 만료시간을 적용하며 클라이언트가 보낸 만료값을 신뢰하지 않는다.
- [ ] 로그아웃 시 서버 측 세션 또는 토큰을 무효화하고 쿠키만 삭제하는 것으로 끝내지 않는다.
- [ ] 로그인 성공, 권한 변경, 비밀번호 변경 후 세션 ID를 회전해 세션 고정을 방지한다.
- [ ] 비밀번호, 이메일, MFA, 결제수단 등 민감정보 변경 전에 현재 비밀번호나 MFA로 재인증한다.
- [ ] 탈퇴, 비밀번호 재설정, 계정 잠금 후 기존 세션을 무효화한다.
- [ ] 만료 또는 무효화된 세션으로 중요 기능을 요청했을 때 서버가 인증 페이지나 401/403으로 처리하는 테스트가 있다.

통과 기준: 오래되거나 탈취된 세션이 무기한 사용되지 않고, 민감 작업은 최근 인증을 거쳐야 한다.

### 1.6 실패 로그인 방어

연계 기준: 반복된 인증시도 제한 기능 부재, 오류 메시지 정보노출

- [ ] 계정과 IP 주소를 함께 고려한 로그인 시도 횟수 또는 속도 제한이 있다.
- [ ] 연속 실패 시 점진적 지연, 일시 잠금, CAPTCHA 또는 추가 인증 중 하나 이상을 적용한다.
- [ ] 잠금 정책 자체가 특정 계정에 대한 서비스 거부 공격 수단이 되지 않도록 해제 절차와 IP 기반 통제를 함께 고려한다.
- [ ] 로그인 실패 응답은 사용자 존재 여부, 잠금 여부, 비밀번호 오류를 과도하게 구분하여 노출하지 않는다.
- [ ] 성공과 실패 이벤트를 보안 로그에 기록하되 입력된 비밀번호와 토큰은 기록하지 않는다.
- [ ] 분산 IP, 동일 계정, 동일 IP의 다수 계정 및 병렬 요청에 대한 제한 테스트가 있다.

통과 기준: 반복·자동화된 인증 시도의 성공 가능성이 제한되고 계정 존재 여부가 응답으로 쉽게 열거되지 않는다.

### 1.7 오류 메시지

연계 기준: 오류 메시지 정보노출, 오류상황 대응 부재, 부적절한 예외 처리, 제거되지 않고 남은 디버그 코드

- [ ] 운영환경에서 Flask/Django `DEBUG`가 비활성화돼 있다.
- [ ] 사용자 응답에 스택 트레이스, 절대 경로, SQL, DB 구조, 환경변수, 비밀키 및 내부 서비스 주소가 포함되지 않는다.
- [ ] 회원가입 중복, 인증 실패 등의 응답이 계정 열거에 필요한 정보를 불필요하게 제공하지 않는다.
- [ ] 예외는 `except:`나 `pass`로 무시하지 않고 예상 가능한 예외별로 실패 처리를 수행한다.
- [ ] 서버 로그에는 추적 가능한 요청 ID와 내부 오류 정보를 남기되 비밀번호, 토큰, 세션 ID, 개인정보를 마스킹한다.
- [ ] 400, 401, 403, 404, 409, 429, 500 등 상황별 오류 처리와 사용자용 일반 메시지가 정의돼 있다.

통과 기준: 사용자는 필요한 최소 정보만 받고, 운영자는 민감정보 없이 원인을 추적할 수 있다.

## 2. 상품 등록 및 관리

### 2.1 폼 입력 검증

연계 기준: 정수형 오버플로우, XSS, 보안기능 결정에 사용되는 부적절한 입력값, 위험한 형식 파일 업로드

- [ ] 제목과 설명은 문자열 타입, 필수 여부, 최소·최대 길이 및 허용 문자 정책을 서버에서 검사한다.
- [ ] 가격과 수량은 문자열 결합이나 부동소수점 임의 변환 대신 `Decimal` 또는 명시된 정수 단위를 사용한다.
- [ ] 가격은 0 이상, 서비스가 정한 최대값 이하이며 `NaN`, 무한대, 음수, 지수 표기 등 허용하지 않는 표현을 거부한다.
- [ ] 카테고리, 거래 상태, 통화 등 열거형 값은 서버의 허용목록에 있는 값만 받는다.
- [ ] 판매자 ID, 승인 상태, 노출 순위 등 서버 관리 필드는 클라이언트 입력으로 설정하지 않는다.
- [ ] 상품 이미지는 확장자, MIME, 파일 시그니처, 크기와 개수를 검증하고 무작위 파일명으로 웹 루트 밖에 저장한다.
- [ ] 경계값, 매우 큰 숫자, 잘못된 타입, 추가 필드 및 악성 파일 업로드 테스트가 있다.

통과 기준: 조작된 가격이나 상태가 저장되지 않고, 허용되지 않은 파일과 범위 밖 입력은 DB 변경 전에 거부된다.

### 2.2 XSS 방어

연계 기준: 크로스사이트 스크립트(XSS), 신뢰되지 않은 URL 주소로 자동접속 연결

- [ ] 상품 제목, 설명, 판매자명 등 사용자 입력은 HTML 텍스트·속성·URL·JavaScript 컨텍스트에 맞게 출력 인코딩한다.
- [ ] 템플릿 자동 이스케이프를 유지하며 `|safe`, `Markup`, `mark_safe`, `innerHTML` 사용 지점을 별도로 검토한다.
- [ ] HTML 입력을 허용해야 하면 검증된 sanitizer와 태그·속성·프로토콜 허용목록을 적용한다.
- [ ] 링크와 이미지 URL은 `javascript:`, `data:` 등 위험한 스킴을 거부하고 허용된 `https` URL만 사용한다.
- [ ] 저장형 XSS 페이로드가 상품 목록, 상세, 검색 결과, 관리자 화면에서 실행되지 않는 테스트가 있다.

통과 기준: 저장된 상품 데이터가 표시되는 모든 화면에서 스크립트, 이벤트 핸들러, 위험한 URL이 실행되지 않는다.

### 2.3 인증된 사용자만 등록

연계 기준: 적절한 인증 없는 중요 기능 허용, 부적절한 인가

- [ ] 상품 생성, 수정, 삭제 엔드포인트 모두 서버 측 인증 데코레이터나 미들웨어로 보호한다.
- [ ] 화면에서 버튼을 숨기는 것만으로 접근을 제한하지 않는다.
- [ ] 인증 정보는 요청 본문의 `user_id`가 아니라 검증된 서버 세션 또는 토큰에서 가져온다.
- [ ] 만료, 위조, 서명 오류 또는 비활성화 계정의 세션·토큰은 거부한다.
- [ ] 비로그인 사용자의 직접 API 호출과 다른 HTTP 메서드를 이용한 우회 요청이 401/403으로 거부되는 테스트가 있다.

통과 기준: 인증되지 않은 요청은 어떤 경로와 메서드로도 상품 데이터를 생성·변경·삭제할 수 없다.

### 2.4 소유자 확인

연계 기준: 부적절한 인가, 보안기능 결정에 사용되는 부적절한 입력값

- [ ] 수정·삭제 대상 조회 시 `product_id`뿐 아니라 현재 인증 사용자의 ID 또는 허용된 관리자 역할을 조건에 포함한다.
- [ ] 요청 본문, 쿠키, hidden field의 `owner_id`, `role`, `is_admin` 값을 인가 근거로 사용하지 않는다.
- [ ] 숫자나 UUID를 바꾸어 다른 사용자의 상품에 접근하는 IDOR 테스트가 있다.
- [ ] 일괄 수정·삭제, 이미지 삭제, 상태 변경 등 보조 엔드포인트에도 동일한 소유권 검사가 적용된다.
- [ ] 인가 실패 시 대상의 존재 여부나 소유자 정보를 과도하게 노출하지 않는다.

통과 기준: 인증된 일반 사용자는 자신이 소유하지 않은 상품을 조회 범위 이상으로 변경하거나 삭제할 수 없다.

### 2.5 데이터 무결성

연계 기준: 경쟁조건(TOCTOU), 오류상황 대응 부재, 부적절한 예외 처리

- [ ] 애플리케이션 검증과 별도로 DB에 `NOT NULL`, 길이, 범위, 외래키, 고유성 등의 제약조건을 적용한다.
- [ ] 상품과 이미지·재고 등 여러 테이블 변경은 하나의 트랜잭션으로 처리하고 실패 시 전체 롤백한다.
- [ ] 중복 등록, 동시 수정, 재고·가격 갱신에는 원자적 쿼리, 행 잠금 또는 낙관적 버전 검사를 사용한다.
- [ ] DB 저장 결과와 영향받은 행 수를 확인하고 실패를 성공으로 응답하지 않는다.
- [ ] 예외 발생 후 불완전한 상품이나 고아 파일이 남지 않는 테스트가 있다.

통과 기준: 부분 저장과 동시성으로 인한 값 손실이 없으며 DB 제약을 우회한 잘못된 상태가 만들어지지 않는다.

## 3. 실시간 채팅 및 메시징

### 3.1 메시지 내용 검증

연계 기준: XSS, 코드 삽입, 포맷 스트링 삽입, 종료되지 않는 반복문 또는 재귀 함수

- [ ] 서버가 메시지 타입, 인코딩, 최소·최대 바이트 길이와 필수 필드를 검사한다.
- [ ] 허용된 텍스트 또는 명시된 메시지 형식 외의 바이너리, 중첩 객체, 과도한 배열과 깊이를 거부한다.
- [ ] 메시지 내용을 `eval()`, `exec()`, 템플릿 소스, 사용자 제어 포맷 문자열로 실행하지 않는다.
- [ ] 메시지를 HTML로 표시할 때 자동 이스케이프하고 링크·마크다운을 지원하면 검증된 sanitizer를 적용한다.
- [ ] 제어문자와 CR/LF가 로그 구조를 위조하지 않도록 로그 필드에 구조화 저장하거나 이스케이프한다.
- [ ] 대용량, 깊게 중첩된 JSON, XSS 페이로드 및 잘못된 UTF-8 테스트가 있다.

통과 기준: 악성 메시지가 브라우저나 서버에서 코드로 실행되지 않고, 과도한 입력으로 자원이 고갈되지 않는다.

### 3.2 사용자 인증

연계 기준: 적절한 인증 없는 중요 기능 허용, 잘못된 세션에 의한 데이터 정보 노출

- [ ] WebSocket 또는 Socket.IO 연결 수립 시 세션이나 토큰의 서명, 만료, 사용자 상태를 서버에서 검증한다.
- [ ] 연결 후 방 입장, 메시지 전송, 파일 전송 등 각 중요 이벤트에서도 사용자와 방의 관계를 인가한다.
- [ ] 클라이언트가 보낸 `user_id`, `room_id`, 역할 값을 인증된 신원으로 신뢰하지 않는다.
- [ ] 사용자별 연결 상태와 메시지를 클래스 변수나 전역 mutable 객체에 혼합 저장하지 않는다.
- [ ] 로그아웃, 계정 정지, 권한 변경 시 기존 연결을 종료하거나 다음 이벤트에서 다시 검증한다.
- [ ] 인증 없는 연결, 만료 토큰, 다른 사용자의 토큰 및 다른 방 입장 시도가 거부되는 테스트가 있다.

통과 기준: 연결 자체와 각 채널 행위가 검증된 사용자 및 허용된 대화방에만 귀속된다.

### 3.3 메시지 검증

연계 기준: 보안기능 결정에 사용되는 부적절한 입력값, 부적절한 인가, `None` 참조

- [ ] 이벤트 이름은 서버가 등록한 허용목록에 있는 값만 처리하고 사용자 값으로 함수나 핸들러를 동적 선택하지 않는다.
- [ ] 페이로드에 JSON Schema, Pydantic, Marshmallow 등 명시적 스키마를 적용한다.
- [ ] 발신자 ID와 권한은 연결의 인증 컨텍스트에서 가져오며 페이로드 값으로 덮어쓰지 않는다.
- [ ] 수신자와 대화방의 존재 여부, 차단 관계, 참여 권한을 메시지 저장·전송 전에 확인한다.
- [ ] 누락 필드와 조회 실패 결과의 `None` 여부를 확인하고 예외로 연결 전체가 중단되지 않게 처리한다.
- [ ] 동일 메시지 재전송에 대한 중복 방지 ID 또는 멱등성 정책이 필요한지 검토한다.

통과 기준: 형식이 맞더라도 권한이 없는 메시지는 저장·전송되지 않으며 잘못된 페이로드는 해당 요청만 안전하게 실패한다.

### 3.4 Rate Limiting

연계 기준: 종료되지 않는 반복문 또는 재귀 함수, 반복된 인증시도 제한 기능 부재

- [ ] 계정, 연결, IP, 대화방별 메시지 전송 횟수와 바이트 수 제한을 설정한다.
- [ ] 짧은 순간의 정상 트래픽은 허용하되 지속적인 초과 요청에는 지연, 폐기 또는 연결 종료를 적용한다.
- [ ] 연결 생성, 방 입장, 타이핑 알림, 파일 업로드 등 메시지 외의 고빈도 이벤트도 제한한다.
- [ ] 서버 큐, 메시지 크기, 대기 시간과 동시 연결 수에 상한을 둔다.
- [ ] 여러 연결과 분산 IP를 이용한 우회, 대용량 메시지 연속 전송 테스트가 있다.
- [ ] 제한 초과 이벤트를 모니터링하되 메시지 본문과 인증정보를 불필요하게 기록하지 않는다.

통과 기준: 한 사용자나 자동화 클라이언트가 채팅 서버의 CPU, 메모리, 큐 또는 네트워크를 독점할 수 없다.

### 3.5 연결 암호화

연계 기준: 암호화되지 않은 중요정보, 부적절한 인증서 유효성 검증, 충분하지 않은 키 길이 사용

- [ ] 운영환경에서 `wss://`만 허용하고 평문 `ws://`로 자동 폴백하지 않는다.
- [ ] TLS 인증서의 신뢰 체인, 유효기간과 호스트명을 검증하며 자체 인증서 무시 옵션을 사용하지 않는다.
- [ ] 역방향 프록시 이후 내부 구간도 위협 모델에 따라 TLS 또는 보호된 네트워크를 사용한다.
- [ ] 인증 토큰을 URL 쿼리 문자열에 넣어 로그나 브라우저 기록에 노출하지 않는다.
- [ ] TLS 종료 프록시의 전달 헤더를 신뢰할 프록시 범위에서만 받아 원래 연결 보안 상태를 판단한다.
- [ ] 유효하지 않은 인증서와 평문 연결이 운영 설정에서 거부되는 테스트가 있다.

통과 기준: 채팅 내용과 인증정보가 전송 중 평문으로 노출되지 않고 클라이언트가 통신 상대의 신원을 검증한다.

## 4. 안전 거래 및 신고

### 4.1 폼 입력 검증

연계 기준: XSS, SQL 삽입, 정수형 오버플로우, 보안기능 결정에 사용되는 부적절한 입력값

- [ ] `target_id`는 요구되는 정수 또는 UUID 형식으로 파싱하고 대상이 실제 존재하는지 확인한다.
- [ ] 신고 사유 코드는 서버 허용목록에서 선택하며 자유 입력 설명에는 최소·최대 길이와 타입 검증을 적용한다.
- [ ] 신고자 ID, 처리 상태, 관리자 메모, 제재 수준은 요청값으로 설정하지 않고 서버가 결정한다.
- [ ] 신고 내용은 파라미터 바인딩으로 저장하고 관리자 화면에 표시할 때 이스케이프한다.
- [ ] 범위 밖 ID, 존재하지 않는 대상, 자기 자신 신고, 과도한 길이, XSS 및 SQL 메타문자 테스트가 있다.

통과 기준: 조작된 식별자와 사유가 저장되지 않고 신고 내용이 사용자·관리자 화면에서 코드로 실행되지 않는다.

### 4.2 인증된 사용자 접근

연계 기준: 적절한 인증 없는 중요 기능 허용, 부적절한 인가, CSRF

- [ ] 신고 생성과 취소는 인증된 사용자에게만 허용한다.
- [ ] 신고 조회·처리·제재 기능은 별도의 관리자 또는 검토자 역할을 서버에서 확인한다.
- [ ] 사용자 신고 목록은 현재 인증 사용자의 신고로 범위를 제한한다.
- [ ] 쿠키 인증을 사용하는 신고 생성·취소 요청에는 CSRF 보호가 적용된다.
- [ ] 일반 사용자의 관리자 API 직접 호출과 다른 사용자의 신고 ID 접근이 403/404로 거부되는 테스트가 있다.

통과 기준: 인증되지 않은 사용자는 신고를 생성할 수 없고 일반 사용자는 관리자 처리 기능이나 타인의 신고에 접근할 수 없다.

### 4.3 데이터 무결성 및 로그 관리

연계 기준: SQL 삽입, HTTP 응답분할, 경쟁조건, 오류상황 대응 부재

- [ ] 신고 저장은 ORM 또는 파라미터 바인딩을 사용하며 동적 SQL 문자열을 만들지 않는다.
- [ ] 신고, 상태 이력, 감사 로그 저장을 필요한 범위에서 트랜잭션으로 처리한다.
- [ ] 감사 로그에 행위자, 대상, 동작, 결과, 서버 시각, 요청 추적 ID를 구조화해 기록한다.
- [ ] 로그에 세션 ID, 토큰, 비밀번호와 불필요한 개인정보를 기록하지 않는다.
- [ ] 사용자 입력의 CR/LF와 제어문자가 로그 라인이나 필드를 위조하지 못하도록 구조화 로깅 또는 이스케이프를 사용한다.
- [ ] 감사 로그는 일반 애플리케이션 사용자가 수정·삭제할 수 없도록 최소 권한과 보존 정책을 적용한다.
- [ ] 로그 저장 실패를 무시한 채 민감한 관리자 처리를 계속할지 정책적으로 결정하고 코드에 반영한다.

통과 기준: 신고 상태와 감사 이력이 일관되며, 로그 변조·삽입과 민감정보 노출 없이 사후 추적이 가능하다.

### 4.4 신고 남용 방지

연계 기준: 반복된 인증시도 제한 기능 부재, 부적절한 인가, 경쟁조건

- [ ] 사용자, IP, 대상별 신고 횟수와 시간당 생성률을 제한한다.
- [ ] 동일 신고자와 대상, 동일 사유의 중복 신고를 DB 고유 제약 또는 원자적 검사로 방지한다.
- [ ] 차단·정지된 계정과 자동화된 클라이언트의 신고 생성 정책을 적용한다.
- [ ] 신고 횟수만으로 자동 제재하지 않고 관리자 검토, 신뢰도 또는 추가 검증 절차를 둔다.
- [ ] 여러 동시 요청으로 제한을 우회하지 못하도록 카운터 갱신을 원자적으로 처리한다.
- [ ] 제한 초과 응답은 429 등 일관된 상태로 반환하며 내부 탐지 기준을 과도하게 노출하지 않는다.

통과 기준: 단일 또는 다중 연결을 통한 반복 신고가 제한되고, 신고 수만 조작해 타인을 자동 제재할 수 없다.

## 5. 관리자 기능

### 5.1 연결 암호화

연계 기준: 암호화되지 않은 중요정보, 부적절한 인증서 유효성 검증, 취약한 암호화 알고리즘 사용, 사용자 하드디스크에 저장되는 쿠키를 통한 정보 노출

- [ ] 운영환경에서 로그인, 회원 제재, 상품 노출 제어, 감사 로그 조회를 포함한 모든 관리자 요청을 HTTPS로만 처리하고 평문 HTTP 요청은 HTTPS로 리다이렉트하거나 거부한다.
- [ ] TLS 1.2 이상과 안전한 암호 스위트를 사용하며 SSL, TLS 1.0/1.1 및 알려진 취약 암호 스위트를 비활성화한다.
- [ ] 서버 인증서의 신뢰 체인, 호스트명, 유효기간을 검증하고 자체 서명 인증서 허용이나 인증서 검증 비활성화 옵션을 운영환경에서 사용하지 않는다.
- [ ] 관리자 세션 쿠키에 `Secure`, `HttpOnly`, 적절한 `SameSite`를 적용하고 필요한 최소 `Domain`, `Path`, 만료 범위만 부여한다.
- [ ] 세션 ID, CSRF 토큰, 관리자 권한 정보와 개인정보를 URL 쿼리 문자열이나 프래그먼트에 포함하지 않는다.
- [ ] TLS 종료 프록시를 사용하는 경우 신뢰된 프록시의 전달 헤더만 사용해 원래 요청 스킴을 판단하고, 프록시와 애플리케이션 사이 구간도 보호된 네트워크 또는 TLS로 구성한다.
- [ ] 관리자 화면이 HTTP 스크립트, 이미지, WebSocket 등 혼합 콘텐츠를 불러오지 않으며 외부 리소스도 신뢰된 HTTPS 출처로 제한한다.
- [ ] HTTPS 응답에 충분히 검증된 `Strict-Transport-Security`를 적용하고 정상·오류 응답 모두 보안 쿠키와 전송 보안 정책을 유지한다.
- [ ] 평문 HTTP, 유효하지 않거나 만료된 인증서, 다른 호스트 인증서 및 변조된 세션 쿠키로 관리자 기능에 접근할 수 없는 통합 테스트가 있다.

통과 기준: 관리자 인증정보, 세션, 회원 및 상품 관리 데이터가 전송 중 평문으로 노출되지 않고 클라이언트가 올바른 서버와 통신함을 검증할 수 있다.

### 5.2 관리자 활동 로그 관리

연계 기준: SQL 삽입, XSS, HTTP 응답분할, 경쟁조건(TOCTOU), 오류상황 대응 부재, 중요한 자원에 대한 잘못된 권한 설정

- [ ] 관리자 권한 부여, 회원 제재·해제, 상품 숨김·복구 등 상태 변경 행위와 회원·제재 이력·상품·감사 로그 조회 행위를 감사 로그에 기록한다.
- [ ] 각 감사 로그에 행위자 ID와 당시 사용자명, 대상 유형과 식별자, 동작, 결과, 서버 기준 UTC 시각, 요청 추적 ID 및 필요한 최소 세부 정보를 구조화해 저장한다.
- [ ] 감사 로그의 행위자 정보는 요청 본문, 쿠키, hidden field의 `user_id`, `username`, `role`이 아니라 검증된 서버 세션과 DB에서 가져온다.
- [ ] 비밀번호, 세션 ID, CSRF 토큰, 쿠키, `Authorization` 헤더, 비밀키와 불필요한 개인정보는 감사 로그에 기록하지 않는다.
- [ ] 사유와 기타 텍스트는 구조화된 필드 또는 JSON으로 저장하고 CR/LF와 제어문자가 로그 항목을 분리하거나 위조하지 못하게 처리한다.
- [ ] 감사 로그를 관리자 화면에 표시할 때 템플릿 자동 이스케이프를 유지하여 사유, 상품명, 사용자명에 포함된 스크립트가 실행되지 않게 한다.
- [ ] 관리자 상태 변경과 해당 감사 로그 생성을 하나의 DB 트랜잭션으로 처리하고 감사 로그 저장 실패 시 상태 변경도 롤백한다.
- [ ] 감사 대상 조회의 로그 저장이 실패하면 민감한 관리자 데이터를 반환하지 않는 등 명시된 실패 정책을 적용한다.
- [ ] 감사 로그는 애플리케이션에서 수정·삭제 기능을 제공하지 않고 DB 권한이나 트리거로 append-only 속성을 보호한다.
- [ ] 감사 로그 조회도 관리자 역할로 제한하고 페이지네이션, 조회 기간 제한 등으로 대량 조회에 따른 자원 고갈을 방지한다.
- [ ] 감사 로그의 보존기간, 백업, 접근권한, 무결성 점검, 이상 행위 모니터링 및 보존기간 종료 후 안전한 파기 절차를 운영 정책으로 정의한다.
- [ ] 정상 변경·조회, 로그 저장 실패, 로그 변조·삭제, XSS·CR/LF 페이로드 및 동시 관리자 요청에 대한 테스트가 있다.

통과 기준: 관리자의 중요 행위를 행위자·대상·시각·결과 단위로 사후 추적할 수 있고, 감사 기록 없이 관리자 변경이나 민감정보 조회가 완료되지 않으며 일반 사용자와 관리자 모두 기존 기록을 변조할 수 없다.

### 5.3 관리자 인증 및 인가

연계 기준: 적절한 인증 없는 중요 기능 허용, 부적절한 인가, 보안기능 결정에 사용되는 부적절한 입력값, CSRF, 잘못된 세션에 의한 데이터 정보 노출

- [ ] 일반 사용자와 관리자 역할을 서버 DB의 허용된 열거값으로 구분하고 신규 회원은 항상 최소 권한인 일반 사용자로 생성한다.
- [ ] 최초 관리자 지정과 권한 변경은 공개 회원가입·프로필 요청에서 분리하며 승인된 운영 설정이나 별도 보호 절차로만 수행한다.
- [ ] `role`, `is_admin`, `user_id` 등 권한 관련 요청 필드는 거부하거나 무시하고 쿠키, 세션의 클라이언트 데이터 또는 hidden field를 인가 근거로 사용하지 않는다.
- [ ] 모든 관리자 화면과 API에서 서버가 검증한 현재 세션의 사용자와 DB 역할을 다시 확인하고 화면에서 메뉴나 버튼을 숨기는 것만으로 접근을 통제하지 않는다.
- [ ] 비로그인 요청은 인증 단계에서 거부하고 로그인한 일반 사용자의 관리자 API 직접 호출은 403 또는 존재 은닉이 필요한 경우 404로 거부한다.
- [ ] 회원 제재·해제는 관리자에게만 허용하며 자기 자신 또는 보호 대상 관리자 계정을 제재하여 관리 기능을 잠그지 못하게 한다.
- [ ] 상품 숨김·복구는 관리자에게만 허용하고 일반 판매자가 상품 수정 요청에 노출 상태나 숨김 사유를 추가해 관리자 결정을 덮어쓰지 못하게 한다.
- [ ] 대상 회원, 제재 및 상품 식별자를 서버에서 조회하고 존재 여부와 현재 상태를 확인하여 IDOR, 존재하지 않는 대상, 이미 처리된 대상 요청을 거부한다.
- [ ] 관리자 상태 변경 요청에 CSRF 토큰과 허용된 `Origin` 검증을 적용하고 `GET` 요청으로 제재·해제·숨김·복구를 수행하지 않는다.
- [ ] 제재 기간, 처리 사유, 노출 상태와 버전 값은 타입, 길이, 범위 및 서버 허용목록으로 검증하며 모든 SQL 값은 파라미터 바인딩으로 전달한다.
- [ ] 동시 제재나 상품 노출 변경에는 트랜잭션, 영향받은 행 수 확인 및 낙관적 버전 검사 등을 적용해 중복 처리와 갱신 손실을 방지한다.
- [ ] 회원 제재와 권한 변경 시 기존 HTTP 세션을 서버에서 폐기하고 연결 중인 WebSocket·Socket.IO 요청도 다음 행위 전에 사용자 상태와 역할을 재검증한다.
- [ ] 만료, 위조, 서버에서 폐기된 세션과 제재 중인 계정은 관리자·일반 기능 모두 이용할 수 없으며 권한 변경 후에는 재로그인을 요구한다.
- [ ] 일반 사용자의 직접 관리자 URL·메서드 호출, 역할 필드 변조, 다른 대상 ID 대입, CSRF 누락·변조, 오래된 버전 및 제재된 세션에 대한 실패 테스트가 있다.

통과 기준: 서버가 인증한 관리자만 회원 제재와 상품 노출 제어를 수행할 수 있고, 일반 사용자는 화면·API·세션·요청값 조작이나 동시 요청으로 관리자 권한 및 처리 결과를 획득하거나 변경할 수 없다.

## 6. 전체 시스템

### 6.1 ORM 및 파라미터 바인딩

연계 기준: SQL 삽입

- [ ] 모든 사용자 입력은 ORM 필터 또는 DB 드라이버의 파라미터 바인딩으로 전달한다.
- [ ] f-string, `%`, `.format()`, 문자열 덧셈으로 SQL을 생성하는 코드가 없다.
- [ ] `raw()`, `text()`, `execute()` 등 원시 SQL 사용 지점을 전수 확인하고 값은 별도 파라미터로 바인딩한다.
- [ ] 테이블명, 열 이름, 정렬 방향처럼 바인딩할 수 없는 SQL 식별자는 고정 매핑 또는 허용목록으로 선택한다.
- [ ] `LIKE`, `IN`, 일괄 업데이트, 검색, 보고서 및 관리자 기능에도 동일한 원칙을 적용한다.
- [ ] 따옴표, 주석, 논리식, 시간 지연 등을 포함한 SQL 삽입 테스트에서 쿼리 구조가 변경되지 않는다.

통과 기준: 외부 입력은 항상 SQL의 데이터 값으로만 처리되며 쿼리 구조나 식별자를 임의로 바꾸지 못한다.

### 6.2 데이터베이스 권한

연계 기준: 중요한 자원에 대한 잘못된 권한 설정, 하드코드된 중요정보

- [ ] 애플리케이션 DB 계정은 필요한 DB, 스키마, 테이블 및 작업에만 권한이 있다.
- [ ] 일반 실행 계정에 DB 관리자, 사용자 생성, 권한 부여, 전체 스키마 변경 권한이 없다.
- [ ] 마이그레이션 계정과 런타임 계정을 분리한다.
- [ ] 읽기 전용 작업은 가능하면 별도 읽기 전용 계정을 사용한다.
- [ ] DB 비밀번호와 인증서는 소스코드나 이미지에 포함하지 않고 승인된 비밀 저장소 또는 보호된 환경 설정에서 주입한다.
- [ ] 비밀 파일과 설정 파일의 OS 권한은 서비스 계정에 필요한 최소 범위로 제한한다.
- [ ] 개발, 테스트, 운영 DB의 계정과 자격증명을 분리한다.

통과 기준: 애플리케이션 계정이 탈취되더라도 영향이 필요한 데이터와 작업 범위로 제한된다.

### 6.3 보안 헤더 설정

연계 기준: XSS, 신뢰되지 않은 URL 주소로 자동접속 연결, HTTP 응답분할

- [ ] `Content-Security-Policy`가 필요한 스크립트·스타일·이미지·연결 출처만 허용하며 불필요한 `unsafe-inline`과 `unsafe-eval`이 없다.
- [ ] 클릭재킹 방지를 위해 CSP `frame-ancestors` 또는 `X-Frame-Options`를 설정한다.
- [ ] `X-Content-Type-Options: nosniff`를 설정한다.
- [ ] `Referrer-Policy`와 필요한 범위의 `Permissions-Policy`를 설정한다.
- [ ] HTTPS 운영환경에서 충분히 검증한 후 `Strict-Transport-Security`를 설정한다.
- [ ] 헤더 값에 사용자 입력을 직접 넣지 않으며 불가피한 값은 CR/LF를 거부하고 허용목록으로 제한한다.
- [ ] 헤더가 애플리케이션과 프록시에서 중복·충돌하거나 특정 오류 응답에서 누락되지 않는지 통합 테스트한다.

통과 기준: 정상 응답과 오류 응답 모두 일관된 보안 헤더를 가지며 사용자 입력으로 응답 헤더를 분할할 수 없다.

### 6.4 HTTPS 적용

연계 기준: 암호화되지 않은 중요정보, 부적절한 인증서 유효성 검증, 취약한 암호화 알고리즘 사용

- [ ] 운영환경의 모든 HTTP 요청을 HTTPS로 리다이렉트하고 인증·개인정보 API는 평문 HTTP로 처리하지 않는다.
- [ ] 유효한 인증서, 신뢰 체인, 호스트명과 만료일을 확인하고 자동 갱신 및 만료 모니터링을 구성한다.
- [ ] 서버와 외부 API 클라이언트에서 인증서 검증 비활성화 옵션을 사용하지 않는다.
- [ ] 오래된 TLS 프로토콜과 취약한 암호 스위트를 비활성화한다.
- [ ] 프록시 환경에서는 신뢰된 프록시의 전달 헤더만 사용해 원래 스킴을 판별한다.
- [ ] HTTP, 유효하지 않은 인증서, 다른 호스트 인증서 및 만료 인증서에 대한 테스트가 있다.

통과 기준: 클라이언트와 서버 및 중요 내부·외부 연동 구간에서 기밀성과 상대방 인증이 유지된다.

### 6.5 에러 및 예외 처리

연계 기준: 오류 메시지 정보노출, 오류상황 대응 부재, 부적절한 예외 처리, 부적절한 자원 해제

- [ ] 예상 가능한 DB, 파일, 네트워크, 파싱 예외를 구체적인 예외 타입별로 처리한다.
- [ ] bare `except:`, 무조건적인 `except Exception: pass`, 실패 후 취약한 기본값 사용이 없다.
- [ ] 실패 시 트랜잭션을 롤백하고 파일, 소켓, DB 연결, 락을 `with` 또는 `finally`로 반환한다.
- [ ] 외부 응답에는 일반화된 오류 코드와 요청 추적 ID만 제공하고 내부 예외 정보는 노출하지 않는다.
- [ ] 로그에서 비밀번호, 토큰, 쿠키, Authorization 헤더, DB 연결 문자열과 개인정보를 제거 또는 마스킹한다.
- [ ] 부분 실패, 타임아웃, 외부 서비스 오류 및 로그 시스템 오류 시의 처리 정책이 테스트돼 있다.

통과 기준: 오류가 보안 통제를 우회하거나 부분 데이터를 남기지 않고, 자원을 누수하지 않으며, 사용자에게 내부 정보를 노출하지 않는다.

### 6.6 라이브러리 및 의존성 관리

연계 기준: 취약한 API 사용, 무결성 검사 없는 코드 다운로드, 하드코드된 중요정보

- [ ] Flask, SQLAlchemy, Flask-SocketIO 및 전이 의존성의 정확한 버전을 lock 파일로 관리한다.
- [ ] 공식 저장소와 신뢰된 인덱스에서만 패키지를 받으며 비슷한 이름의 패키지와 출처를 확인한다.
- [ ] CI에서 알려진 취약점 검사를 실행하고 심각도 기준에 따라 빌드 실패 또는 예외 승인 절차를 적용한다.
- [ ] 알려진 CVE가 있는 버전은 수정 버전으로 업데이트하고, 즉시 업데이트할 수 없으면 영향 분석과 임시 완화책을 기록한다.
- [ ] 사용하지 않거나 유지보수가 중단된 패키지는 제거 또는 대체한다.
- [ ] 가능하면 패키지 해시나 서명을 검증하고 다운로드한 코드를 무결성 검사 없이 실행하지 않는다.
- [ ] 패키지명, 공급자, 버전, 해시, 라이선스, 의존 관계를 포함하는 SBOM을 생성하고 배포본과 함께 최신화한다.
- [ ] 소스, 설정, 테스트 데이터와 패키지 설정에 API 키나 저장소 자격증명이 포함되지 않는다.

통과 기준: 배포본에 포함된 모든 직접·전이 의존성을 식별할 수 있고, 신규 취약점 발견 시 영향받는 버전을 찾아 검증된 업데이트로 교체할 수 있다.

## 7. 상품 검색 기능

### 7.1 검색 키워드 확인 및 XSS 방어

연계 기준: 크로스사이트 스크립트(XSS), SQL 삽입, 보안기능 결정에 사용되는 부적절한 입력값, 오류 메시지 정보노출

- [ ] 검색 키워드는 쿼리 파라미터, 폼, JSON 등 입력 경로와 관계없이 서버에서 문자열 타입, 허용된 최소·최대 길이, 문자 인코딩 및 제어문자 포함 여부를 검증하고 일관된 방식으로 정규화한다.
- [ ] 키워드에 `<`, `>`, 따옴표 등이 있다는 이유만으로 XSS 방어가 완료됐다고 판단하지 않으며, 검색어가 검색창의 `value`, 검색 결과 제목, 결과 없음 안내, 페이지 제목 등 HTML에 다시 표시될 때 각 출력 컨텍스트에 맞게 이스케이프한다.
- [ ] Jinja/Django 템플릿의 자동 이스케이프를 유지하고 검색 키워드 또는 키워드가 포함된 상품 정보를 `|safe`, `Markup`, `mark_safe`나 HTML 문자열 결합으로 출력하지 않는다.
- [ ] 클라이언트에서 검색어, 자동완성 결과 또는 강조 표시를 렌더링할 때 `innerHTML`, `outerHTML`, `insertAdjacentHTML`을 사용하지 않고 `textContent` 등 텍스트 전용 API를 사용한다. HTML 강조 표시가 불가피하면 검증된 sanitizer와 엄격한 허용목록을 적용한다.
- [ ] 검색 API가 JSON을 반환하는 경우 올바른 `Content-Type`을 설정하고, 클라이언트가 응답 값을 HTML 코드로 재해석하지 않도록 한다.
- [ ] 검색 쿼리는 문자열 결합, f-string, `%`, `format()`으로 SQL을 조립하지 않고 파라미터 바인딩 또는 ORM을 사용한다. `LIKE` 검색의 `%`, `_`를 일반 문자로 취급해야 한다면 별도의 이스케이프 규칙과 `ESCAPE` 절을 일관되게 적용한다.
- [ ] 검증에 실패한 검색어는 DB 조회와 템플릿 렌더링 전에 400 등으로 거부하며, 오류 응답에 원본 공격 문자열, SQL, 스택 트레이스 또는 내부 경로를 그대로 포함하지 않는다.
- [ ] `<script>alert(1)</script>`, `"><img src=x onerror=alert(1)>`, `javascript:` URL, HTML 엔티티·유니코드 변형 및 최대 길이 초과 입력을 검색 화면, 자동완성, 결과 없음 화면과 오류 화면에 전달해 스크립트가 실행되지 않는 테스트가 있다.

통과 기준: 공격자가 조작한 검색 키워드와 검색 결과 데이터가 어느 출력 경로에서도 실행 가능한 HTML·JavaScript로 해석되지 않고, 검색 쿼리 구조나 오류 응답을 통해 서버 내부정보에 영향을 주거나 이를 노출하지 않는다.

### 7.2 검열된 상품에 대한 인증 및 객체 단위 인가

연계 기준: 적절한 인증 없는 중요 기능 허용, 부적절한 인가, 보안기능 결정에 사용되는 부적절한 입력값, 경쟁조건(TOCTOU), 잘못된 세션에 의한 데이터 정보 노출

- [ ] 관리자가 숨김·검열한 상품은 기본 검색 결과, 추천·인기 목록, 자동완성, 필터별 개수와 전체 건수에서 제외하며, 판매자인 게시자와 권한 있는 관리자에게만 정책에 따라 별도로 노출한다.
- [ ] 숨김 상품의 상세 페이지, 검색 API, 이미지·첨부파일, 미리보기 및 공유 링크 등 모든 조회 경로에서 현재 인증 사용자가 판매자인지 또는 허용된 관리자인지를 서버에서 객체 단위로 확인한다.
- [ ] 소유권과 관리자 권한은 요청의 `user_id`, `seller_id`, `owner_id`, `role`, `is_admin`, 쿠키나 hidden field가 아니라 검증된 서버 세션·토큰의 사용자 ID와 DB의 최신 상품·역할 상태를 기준으로 결정한다.
- [ ] 화면에서 검색 결과나 링크를 숨기는 것만으로 접근을 제한하지 않으며, 비로그인 사용자와 다른 일반 사용자가 상품 ID·UUID를 추측하거나 URL과 API 파라미터를 변경해도 검열된 상품에 접근할 수 없다.
- [ ] 인가되지 않은 요청에는 정책에 따라 403 또는 상품 존재 여부를 숨기는 404를 반환하고, 제목, 설명, 가격, 판매자, 숨김 사유, 이미지 경로 등 상품 정보가 응답 본문·헤더·오류 메시지에 포함되지 않는다.
- [ ] 검색 결과 캐시, CDN, 검색 인덱스와 애플리케이션 캐시에도 공개 범위가 반영되고, 상품이 숨김 처리되면 기존 공개 캐시와 인덱스를 즉시 또는 정의된 최대 지연시간 안에 무효화한다. 권한별 응답을 캐시한다면 사용자·역할 범위가 다른 요청 사이에 결과를 공유하지 않는다.
- [ ] 검색 결과 조회 후 상세 페이지나 이미지 요청 전에 상품이 숨김 처리되는 경우를 대비해 각 자원 제공 시점에 최신 공개 상태와 접근 권한을 다시 확인한다.
- [ ] 비로그인 사용자, 다른 일반 사용자, 상품 판매자, 권한 있는 관리자 각각에 대해 검색 목록, 직접 상세 URL, 검색 API, 이미지 URL 및 캐시된 응답의 접근 허용·거부를 검증하는 통합 테스트가 있다.

통과 기준: 검열된 상품은 검색 및 연관 조회 경로에서 일반 사용자에게 존재나 내용이 노출되지 않으며, 서버가 확인한 게시자와 권한 있는 관리자만 최신 상품 상태를 기준으로 접근할 수 있다.

## 8. 유저 간 송금 기능

### 8.1 연결 암호화 및 거래정보 비공개

연계 기준: 암호화되지 않은 중요정보, 부적절한 인증서 유효성 검증, 부적절한 인가, 잘못된 세션에 의한 데이터 정보 노출, 사용자 하드디스크에 저장되는 쿠키를 통한 정보 노출

- [ ] 운영환경의 상품 구매, 거래내역 조회, 구매확정, 취소·환불 및 정산 요청을 HTTPS로만 처리하고 평문 HTTP 요청을 거부하거나 HTTPS로 전환한다.
- [ ] TLS 1.2 이상과 안전한 암호 스위트를 사용하고, 인증서의 신뢰 체인·호스트명·유효기간 검증을 비활성화하지 않는다.
- [ ] TLS 종료 프록시를 사용하면 신뢰할 프록시 범위의 전달 헤더만 사용해 원래 요청 스킴을 판단하고, 프록시와 애플리케이션 사이도 TLS 또는 보호된 네트워크로 구성한다.
- [ ] 세션 쿠키에 `Secure`, `HttpOnly`, 적절한 `SameSite`를 적용하고 거래 당사자, 잔액, 주문·결제 상태를 쿠키에 저장하지 않는다.
- [ ] 거래 목록·상세·상태 변경 API는 검증된 서버 세션의 사용자 ID와 DB의 구매자·판매자 ID를 비교하여 당사자에게만 허용하고, 요청 본문·쿠키·hidden field의 `buyer_id`, `seller_id`, `user_id`, `role`을 인가 근거로 사용하지 않는다.
- [ ] 사용자가 주문 ID를 변경하거나 다른 사용자의 거래 URL에 직접 접근해도 정책에 따라 404 또는 403으로 거부하고 상품명, 가격, 구매자, 판매자, 잔액과 결제 상태를 응답에 포함하지 않는다.
- [ ] 거래 당사자·상품·금액을 URL 쿼리 문자열, `Referer`, 분석 이벤트, 접근 로그 및 사용자용 오류 메시지에 불필요하게 노출하지 않으며 로그에는 요청 ID와 최소한의 식별자만 남긴다.
- [ ] 거래 페이지와 API 응답에 `Cache-Control: no-store` 등을 적용하고, CDN·프록시·브라우저 캐시가 다른 사용자의 주문 내용을 재사용하지 않도록 한다.
- [ ] HTTP 캡처 테스트에서 외부 네트워크 관찰자가 TLS 암호문 외의 애플리케이션 데이터를 확인할 수 없고, 비당사자 세션으로 목록·상세·직접 URL·상태 변경 요청이 모두 거부되는 통합 테스트가 있다.

통과 기준: 외부 네트워크 관찰자는 전송 내용에서 구매자·판매자·상품·금액을 확인할 수 없고, 거래 비당사자는 암호화된 연결을 사용하더라도 주문 객체에 접근하거나 거래 여부를 확인할 수 없다. 단, IP·접속 시간·트래픽 크기 등 TLS가 숨기지 않는 통신 메타데이터는 별도 위협 모델로 평가한다.

### 8.2 가격·송금액 서버측 입력 검증

연계 기준: 정수형 오버플로우, 보안기능 결정에 사용되는 부적절한 입력값, SQL 삽입, HTTP 응답분할, 오류상황 대응 부재

- [ ] 결제 금액은 클라이언트가 보낸 `price`, `amount`, hidden field 또는 화면에 표시된 값을 신뢰하지 않고, 결제 직전에 서버가 DB의 최신 상품 가격을 조회해 결정한다.
- [ ] 주문에는 결제 시점의 상품명과 가격 스냅샷을 저장하고, 이후 상품 가격이 변경돼도 기존 결제·환불·정산 금액은 주문 스냅샷을 기준으로 한다.
- [ ] 가격과 송금액은 원·포인트 등 명시된 최소 단위의 정수로 처리하고, 부동소수점 연산으로 잔액·환불액을 계산하지 않는다.
- [ ] 송금액을 입력받는 API가 추가되면 단일 필드인지 확인한 뒤 ASCII 10진 숫자만 허용하고, 빈 값, 공백, `+`·`-` 부호, 소수, 천 단위 구분자, 지수 표기, `NaN`, 무한대, 유니코드 숫자, 제어문자와 중복 필드를 거부한다.
- [ ] 결제·송금액은 0 이상 서비스 최대값 이하인지 정수 변환 전·후에 검사하고, 자릿수가 과도한 문자열과 DB·런타임 정수 범위를 넘는 값을 거부한다.
- [ ] 구매자·판매자 ID, 수수료, 잔액, 예치·정산 금액, 결제·주문 상태는 클라이언트가 직접 설정하지 못하게 하고 인증 세션, 주문 스냅샷, 서버 정책과 DB의 최신 상태로부터 계산한다.
- [ ] 잔액 차감 전에 구매자 잔액이 결제액 이상인지 확인하고, 일반 사용자와 시스템 예치 계정의 잔액이 음수가 되지 않도록 DB 제약조건과 원자적 갱신으로 보장한다.
- [ ] 가격·금액·상태·ID를 SQL 문자열에 연결하지 않고 모든 조회와 갱신에 파라미터 바인딩 또는 ORM을 사용한다.
- [ ] DB에도 금액의 정수 타입, 최소·최대 범위, 상태 허용목록, 필수값, 외래키와 고유성 제약을 적용해 애플리케이션 검증을 우회한 비정상 데이터를 거부한다.
- [ ] 부족한 잔액, 범위 초과, 잘못된 형식, 조작된 가격, 중복 필드와 상태 불일치는 DB 변경 전에 4xx로 거부하고 사용자 응답에 잔액, SQL, 스택 트레이스 및 내부 계정 식별자를 노출하지 않는다.
- [ ] 최솟값·최댓값, 0원, 잔액과 동일한 금액, 잔액을 1 초과한 금액, 음수, 소수, 지수 표기, 매우 긴 숫자, 변조된 상품 가격을 대상으로 서버측 통합 테스트가 있다.

통과 기준: 클라이언트가 가격·송금액·잔액·수수료를 조작해도 서버가 신뢰한 상품·주문 데이터와 명시적 정수 규칙으로 결제액을 결정하며, 범위 밖 금액이나 잔액 부족 요청은 어떤 원장·주문·상품 변경도 남기지 않는다.

### 8.3 경쟁 상태 방지 및 멱등성 보장

연계 기준: 경쟁조건(검사시점과 사용시점), 오류상황 대응 부재, 부적절한 예외 처리, 보안기능 결정에 사용되는 부적절한 입력값

- [ ] 상품의 판매 상태·가격·판매자, 구매자 잔액과 활성 주문 여부를 검사한 뒤 결제하는 전체 과정을 하나의 DB 트랜잭션 및 필요한 잠금 범위 안에서 실행해 TOCTOU를 방지한다.
- [ ] 주문 생성, 구매자 차감, 시스템 계정 예치, 결제·원장 기록과 상품 예약 상태 변경은 모두 성공하거나 모두 롤백되며 일부만 반영된 중간 상태를 남기지 않는다.
- [ ] 구매확정은 예치 차감, 판매자 지급, 주문·결제 정산 완료와 상품 판매 완료를, 취소는 예치 차감, 구매자 환불, 주문·결제 환불 완료와 상품 판매 재개를 각각 하나의 DB 트랜잭션으로 처리한다.
- [ ] 주문·결제·상품·계정에 버전 번호 또는 동등한 조건부 갱신을 적용하고, 읽은 후 다른 요청이 상태를 바꾸었으면 stale 요청을 409 등으로 거부한다.
- [ ] 명시적 상태 머신으로 허용된 전이만 수행하고, 예를 들어 `paid → settled` 또는 `paid → cancelled` 외의 전이, 최종 상태의 재정산·재환불 및 구매확정 후 취소를 서버와 DB 제약에서 거부한다.
- [ ] 상품별 활성 주문을 하나로 제한하는 DB 고유 제약이나 동등한 원자적 제어를 적용해, 두 구매자가 동시에 결제해도 하나의 주문만 성공하도록 한다.
- [ ] 구매 요청에 충분히 예측 불가능한 멱등성 키를 사용하고 키, 요청자, 상품, 주문과 결과를 서버에 연결하여 중복 클릭·재시도·응답 유실이 중복 차감이나 중복 주문을 만들지 않게 한다.
- [ ] 같은 사용자가 같은 상품에 같은 멱등성 키를 재사용하면 기존 주문 결과를 반환하고, 다른 사용자·상품·요청 내용에 같은 키를 사용하면 충돌로 거부한다.
- [ ] 정산·환불은 주문에서 파생한 고정 멱등성 키 또는 고유 제약을 사용해 같은 요청이 재실행돼도 원장 거래와 잔액 변경이 한 번만 생성되도록 한다.
- [ ] 원장 거래는 수정·삭제하지 않고 차감·예치·환불·정산을 새 행으로 추가하며, 각 거래의 차변·대변 합계가 0이고 계정 잔액과 원장 합계가 일치하는지 검증한다.
- [ ] 두 구매자의 동시 구매, 구매 버튼 연타, 같은 멱등성 키 재전송, 다른 내용의 키 재사용, 구매확정·취소 동시 요청, 정산·환불 재시도와 stale 버전을 병렬 통합 테스트로 검증한다.
- [ ] 트랜잭션 충돌, 잠금 타임아웃, DB 오류 및 중복 키 충돌을 구체적인 예외로 처리하고, 실패 후 자동 재시도 여부와 횟수를 멱등성 보장 범위 안에서 명시한다.

통과 기준: 동일 상품·주문에 대한 병렬 요청과 재시도가 발생해도 구매자 차감, 시스템 예치, 판매자 정산과 구매자 환불은 각 허용된 거래당 한 번만 반영되고, 상품·주문·결제·원장 상태가 항상 일치한다.
